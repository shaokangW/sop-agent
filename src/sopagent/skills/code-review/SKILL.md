# 代码审查

## 工作流
1. **拿 diff**:`bash git diff HEAD` 或 `git diff main...HEAD`(如果是 PR)
2. **理解意图**:读 PR/diff 描述,弄清这次改动想解决什么
3. **读全貌**:对改动的文件用 `read_file` 看完整上下文(diff 只看局部易误判)
4. **逐文件审**,按下面清单过:
   - 正确性:逻辑是否对?边界条件?off-by-one?空值/异常路径?
   - 安全:用户输入是否校验?有无注入/越权/信息泄露?密钥硬编码?
   - 错误处理:异常是否吞了?资源是否释放?
   - 并发:共享状态有无竞态?
   - 可读性:命名是否达意?函数是否过长?注释是否解释 why 而非 what?
   - 测试:新逻辑有无测?测的是行为还是实现?
5. **输出 review**:分级列出
   - 🔴 阻塞(必须改):正确性/安全/数据丢失
   - 🟡 建议(应该改):可读性/测试缺失/设计
   - 🟢 可选(可以改):风格/微优化
   每条给 文件:行号 + 问题 + 建议

## 注意
- 不要只挑风格问题凑数;重点在正确性和安全
- 不确定的地方标出来问作者,别误判
